# facebook_marketplace_search — v1 Spec

Status: Draft v1
Owner: product-manager
Audience: architect (#2), qa-tester (#3), developer (#4)

---

## 1. Problem statement

Facebook Marketplace's native search treats numeric size tokens (e.g. `11` in `gants hockey 11`) as loose keywords, so it cheerfully returns size 7, 14, and 15 listings alongside (or instead of) actual size-11 results. The same fuzziness leaks into other "filter-shaped" criteria: distance and "listed within" are advisory, condition is often mis-tagged by sellers, and price filters miss listings whose price lives in the description rather than the price field. The result is a UI where the user must manually re-filter every page of results, which is exactly the work this product is designed to eliminate. v1's reason to exist is **post-search filter validation**: take the same query a human would type, run it on Marketplace, then drop every returned listing whose actual attributes (parsed from title/description/structured fields) violate the user's stated filters — so the user only ever sees listings that genuinely match.

## 2. Goals & non-goals

**Goals (v1)**
- Single-user, single-machine local CLI tool.
- Accept one free-form query line and parse it into structured filters.
- Run a real Marketplace search and harvest the result list.
- Persist every seen listing in local SQLite so re-runs can diff vs. last run.
- Re-validate every listing's actual attributes against the parsed filters and drop violators *before* showing results to the user.
- Make the validation pass auditable: for each dropped listing, record which filter it failed and why.

**Non-goals (v1)** — explicitly out of scope:
- Auto-messaging the seller / drafting outreach.
- Multi-account or account rotation.
- Background scheduling, cron, or daemonization (one shot per invocation).
- Push/email/desktop notifications for new matches.
- A web or GUI dashboard (CLI only).
- Sponsored / commercial listings or Marketplace Catalog API.
- Categories outside the user's chosen keyword query (no cross-category browsing).

## 3. The wedge — what "validation" actually means

A "filter" is something we can verify against the listing's data after the page is fetched. v1 supports five validators:

| Filter | Source of truth on listing | Validation rule |
|---|---|---|
| **size** (e.g. `11`, `XL`, `size 11"`) | Title + description tokens | Listing must contain the exact size token, bounded by non-digit chars (so `11` does NOT match `110`, `115`, `7-11 set`, etc.). Numeric sizes use word-boundary regex; alpha sizes (`XL`, `M`) use case-insensitive whole-token match. |
| **price range** `min-max` (CAD) | Listing's structured price field | `min ≤ price ≤ max`. Listings with no parseable price are dropped (logged as `no_price`). |
| **distance km** | Listing's distance-from-user field returned by Marketplace | `distance_km ≤ filter_km`. If Marketplace did not return a distance for a listing, drop it (logged as `no_distance`). |
| **listed within N days** | Listing's `created_at` / "listed N days ago" field | `now - listed_at ≤ N days`. Treat "just listed" / "X hours ago" as 0 days. Missing timestamp → drop (`no_listed_at`). |
| **condition** (`new`, `used-like-new`, `used-good`, `used-fair`) | Listing's structured condition field | Exact match against the structured field. v1 does NOT try to infer condition from free text. Missing condition → drop (`no_condition`) only if the user specified one. |

Anything else the user types that we don't recognize as a filter is treated as a free-text **keyword** and passed through to Marketplace's search box verbatim. We do not post-validate keyword matches.

## 4. User stories & acceptance criteria

### Story 1 — Query input

**As a** user
**I want** to type one query line that mixes keywords and filters
**So that** I don't have to click through Marketplace's UI

Example input: `gants hockey 11", new, 10km, $50-100, listed in 1 week`

**Acceptance criteria**
- [ ] CLI accepts the query as a single quoted argument or via stdin.
- [ ] The parser recognizes (in any order, separated by commas or spaces):
  - size token: bare digit (`11`), digit+inch (`11"`, `11 inch`), or alpha (`XL`, `M`, `L`, `S`, `XS`, `XXL`)
  - price range: `$50-100`, `50-100$`, `between 50 and 100`, `under 100`, `over 50`
  - distance: `10km`, `10 km`, `within 10 km`
  - recency: `listed in 1 week`, `last 7 days`, `past 24h`, `today`
  - condition: literal tokens `new`, `like new`, `used`, `good`, `fair`
- [ ] Tokens consumed by filters are stripped; the remainder is the keyword string sent to Marketplace.
- [ ] If parsing is ambiguous, the CLI prints the parsed filter set and asks for confirmation (y/N) before running.

### Story 2 — Run the search

**As a** user
**I want** the tool to execute the search on Marketplace using the parsed filters
**So that** Marketplace pre-narrows results before our validation pass

**Acceptance criteria**
- [ ] Tool sets every Marketplace-supported filter that maps to a parsed filter (price range, distance, condition, days-since-listed) via Marketplace's own controls.
- [ ] Keyword string (with size tokens left in) is submitted to Marketplace's search box.
- [ ] Tool harvests the first N pages of results (default N=3, configurable via `--pages`).
- [ ] For each result, captures: marketplace listing id, URL, title, description (first paragraph if available without a second click), price, currency, location, distance_km, listed_at, condition, seller_id, image URL, raw HTML/JSON snapshot for audit.
- [ ] If Marketplace returns zero results, the tool exits cleanly with a "no results" message — no crash, no empty DB write.

### Story 3 — Local SQLite persistence

**As a** user
**I want** every listing the tool has ever seen stored locally
**So that** I can re-run the search later and see what's new

**Acceptance criteria**
- [ ] On first run, tool creates `~/.fb_marketplace_search/db.sqlite` (path configurable).
- [ ] Schema (at minimum):
  - `searches(id, query_text, parsed_filters_json, run_at)`
  - `listings(marketplace_id PK, url, title, description, price, currency, location, distance_km, listed_at, condition, seller_id, image_url, raw_blob, first_seen_at, last_seen_at)`
  - `search_results(search_id FK, listing_id FK, position, validated_pass BOOL, validation_failures_json)`
- [ ] Inserts are idempotent on `marketplace_id`; re-seeing a listing updates `last_seen_at` and any changed fields (price drops are common and worth tracking).
- [ ] Listings dropped by post-validation are still persisted, but `validated_pass = false` and the failure reasons are recorded.

### Story 4 — Post-search validation

**As a** user
**I want** the tool to drop listings whose actual attributes violate my filters
**So that** the size-11 query doesn't show me size 7 / 14 / 15

**Acceptance criteria**
- [ ] After harvest, the tool runs every listing through the validators in §3.
- [ ] A listing fails validation if **any one** filter rejects it. Failures are AND-combined.
- [ ] For each failed listing, record `{filter: "size", reason: "title contains '14' not '11'"}` in `validation_failures_json`.
- [ ] Default CLI output shows only `validated_pass = true` listings, sorted by position.
- [ ] `--show-rejects` flag prints the dropped listings and their failure reasons (for debugging the validator, and for the success-metric dashboard below).
- [ ] Size validator MUST NOT match substrings: `11` does not satisfy `110`, `115`, `7-11`, `s/m/l/xl/11`, etc. Token must be bounded by non-digit characters or string boundaries.

### Story 5 — Re-run and diff

**As a** user
**I want** to re-run the same query later and see what changed
**So that** I can spot new listings without re-reading the whole list

**Acceptance criteria**
- [ ] Re-running an identical query (same parsed-filters JSON) computes a diff against the most recent prior `searches` row with the same parsed filters.
- [ ] Diff output buckets results into: `NEW` (not in any prior run), `STILL_THERE` (seen in last run too), `GONE` (in last run, not this one), `PRICE_CHANGED` (still there, but price differs).
- [ ] `--only new` flag restricts CLI output to `NEW`.
- [ ] Diff is computed only across listings that pass validation in both runs (so a listing that newly drops below size threshold doesn't show as `GONE` — it's just filtered out).

## 5. Success metric

**Primary metric:** percentage of listings shown to the user (i.e. `validated_pass = true`) that genuinely match every active filter, measured by spot-checking from a held-out manual review of N=20 listings per filter type.

| Filter | Target |
|---|---|
| size | **100%** (this is the wedge — anything less means we shipped the same broken UX as Marketplace) |
| price range | ≥ 99% (small slack for Marketplace returning a stale price field) |
| distance | ≥ 99% |
| listed-within-N-days | ≥ 95% (Marketplace's "X days ago" rounding is coarse) |
| condition | ≥ 90% (sellers mistag; we trust the structured field, not free text) |

**Secondary metrics** (logged, not gated):
- False-negative rate: how many listings the validator dropped that a human reviewer thinks should have passed. Track via `--show-rejects` + manual labeling.
- Pre-validation noise rate: `1 - (validated_pass / total_returned)`. Higher is fine — it means Marketplace is leaky and we're earning our keep.
- Time-to-results: median end-to-end seconds from CLI invocation to printed list. Soft target < 60s for default `--pages 3`.

## 6. Open questions for the architect

These are the decisions I want #2 to answer before #4 starts coding:

1. **Login required?** Marketplace shows partial results to logged-out browsers but gates location-precise distance and full descriptions behind login. Do we (a) require the user to bring a logged-in session/cookie, (b) drive Playwright through a one-time interactive login that persists state, or (c) try to live without login and accept reduced fidelity? My preference: **(b)** — one-time login, persist storage state to disk, like `appointmaker/` does for Costco.
2. **Anti-bot strategy.** Marketplace fingerprints aggressively. Do we go with vanilla Playwright + storage state (simpler), or Camoufox / patchright like `jobhuntingagent/`? What's our retry/backoff policy on a 429 or a "please log in again" wall?
3. **Headless vs. headful.** Headful is friendlier to the user during the one-time login and less likely to trip detection, but blocks the terminal window. Headless is invisible but more detectable. My preference: headful for the login step, headless for the actual search runs unless `--debug` is passed.
4. **Pagination.** Does Marketplace's web UI use infinite scroll, numbered pages, or "load more"? This affects how `--pages` is implemented and whether we can deterministically request page N.
5. **Description harvesting cost.** The result grid shows a truncated title. Some validators (size especially) need the description, which lives behind a click into the listing detail page. Is description-fetching part of the harvest loop, or a lazy second pass only on listings that pass the cheap title-only validators? My preference: **lazy second pass** — fetch description only when title alone doesn't conclusively pass/fail size.
6. **Schema migrations.** v1 is single-user and the DB is throwaway-friendly, but if we add a column in v1.1, do we want Alembic-style migrations or a "blow away the DB" policy? My preference: blow away (it's a cache, not a system of record) — but call it out in the architecture doc.
7. **Rate limiting.** What's a polite cadence for re-running the same query? Marketplace will notice if we hammer it. Do we need a minimum interval between runs of the same query?

## 7. Glossary

- **Listing**: a single item-for-sale page on Marketplace.
- **Filter**: a structured constraint parsed from the user's query that we can verify against listing attributes.
- **Keyword**: free-text portion of the query passed straight to Marketplace's search box; not post-validated.
- **Validator**: the function that decides whether a listing satisfies one specific filter.
- **Validation pass**: the post-harvest step that runs every active validator against every harvested listing.

## 8. Addendum — rulings on architect & QA questions

These resolve the ambiguities raised in `docs/architecture.md` §7 and `docs/test-plan.md` §10. Each is normative — developer and qa-tester should treat them as part of the spec.

### 8.1 From architect (architecture.md §7)

1. **Re-validation on every re-run — CONFIRMED.** Every run revalidates every harvested listing from scratch. Prior `validated_pass` values are not trusted as cache. A validator bugfix takes effect immediately on the next run.
2. **Distance display unit — CONFIRMED.** CLI output always renders distances in km. No miles, no mixed-unit fallback.
3. **Currency assumption — CONFIRMED.** v1 assumes the user's filter is in CAD. If a listing's currency is non-null and not CAD, the price validator fails it with `validation_failures_json` reason `currency_mismatch`. v1 does no FX conversion. (Same ruling as 8.4 below — this is the canonical statement.)
4. **Ambiguous parse trigger — CONFIRMED with the explicit rule below** (also resolves QA AT-1.11 / §10 item 5).
5. **`--only new` on first-ever run — CONFIRMED.** Print every passing listing and a one-line note: `first run for these filters; nothing to diff against yet.` Do not bucket as `NEW` and do not error.
6. **Within-run duplicates — CONFIRMED.** `search_results` collapses dups via the `(search_id, listing_id)` PK with `INSERT OR IGNORE`; the earliest `position` wins. Log once per run as `dedup: dropped N within-run duplicate listings`. (Closes QA DIFF-7 / §10 item 3.)
7. **Re-run rate limiting — CONFIRMED.** Default 5-minute minimum interval keyed on `parsed_filters_hash`. `--min-interval <seconds>` overrides; `--force` bypasses. On refusal: exit non-zero with `Last run was N seconds ago; minimum interval is M. Use --force to override.` (Closes QA RATE-1 / §10 item 4.)
8. **Currency mismatch ruling — CONFIRMED.** See 8.3 above; this is the same ruling, restated so QA's PRICE-10 has one place to point. (Closes QA §10 item 1.)

### 8.2 From QA (test-plan.md §10) — product-scope rulings

**RECENT-8 — "a week ago" rounds to 7 days (PASS for `recency_days=7`).**

> **Why:** spec §5 already concedes that Marketplace's "X days/weeks ago" strings are coarse and sets the recency accuracy target at 95% precisely to absorb this fuzziness. Forcing "a week" to round up to 8 would systematically reject listings the user mentally counts as "within a week", which is the opposite of what the user typed in their query.
>
> **How to apply:** the recency parser maps the literal string `a week ago` (and equivalents: `1 week ago`, `last week`) to `listed_at = now − 7 days`. Plural week strings (`2 weeks ago`) round to `n × 7` days. Day-grain strings (`X days ago`) take their literal value. `just listed`, `today`, sub-day strings (`X hours ago`, `X minutes ago`) all map to 0 days. Anything not on this list → `listed_at = NULL` → fails recency validator with `no_listed_at` per spec §3.

**AT-1.11 — definition of "ambiguous parse" (the y/N confirmation trigger).**

A query is ambiguous, and the CLI MUST print the parsed filter set and prompt `(y/N)` before running, if **any** of the following is true:

1. **Duplicate filter type:** two or more tokens parsed into the same filter type (e.g. two distance tokens like `10km within 5 km`, two prices, two sizes). Pick the first match for the parsed-filter set shown to the user; let them confirm or abort.
2. **Token-class collision:** a token matched a filter regex but its surrounding context is also plausibly keyword (e.g. `new` in `new york yankees`, `large` in `large lot`, `S` as a single letter when it could be an initial). Heuristic: a single-letter alpha-size token (`S`, `M`, `L`) flanked on both sides by alphabetical tokens — not by `size`, `taille`, `:`, `,`, or string boundary — is ambiguous.
3. **Bare-integer collision:** a bare integer (no inch mark, no `size`/`taille` cue) that the parser would consume as a numeric size, when the same integer could plausibly be a quantity, year, or model number (e.g. `iphone 11`, `2 of them`, `model 700`). Concretely: a bare integer is ambiguous unless it is preceded by `size`/`taille`/`sz`/`sz.` OR followed by `"`/`inch`/`in.`.

If none of these triggers fire, the parse is unambiguous and the CLI runs without prompting.

**Escape hatch:** `--assume-yes` (or `-y`) skips the prompt and accepts the parser's first-match interpretation. Intended for scripted runs and the smoke test suite. The chosen interpretation is still echoed to stdout so it's auditable.

**Closes:** test-plan.md §10 items 2 (RECENT-8) and 5 (AT-1.11).
